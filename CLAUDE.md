# Dota 2 GSI — предметы игроков (спектатор)

Контекст-handoff из чата на claude.ai. Цель проекта и принятые решения ниже —
читай это как память проекта.

## Цель
Парсить лайв про-матчи Доты (легальный спектатор-доступ) и доставать предметы
всех игроков обеих команд для аналитики/оверлея.

## Выбранный подход: GSI (Game State Integration)
Официальный механизм Valve. Клиент Доты сам шлёт JSON на локальный HTTP-сервер.
Выбран вместо парсинга демок (manta/clarity), потому что:
- меньше кода, данные уже в JSON;
- не нужно возиться с TV-broadcast протоколом и резолвом item-хэндлов.

Демки имело бы смысл парсить только ради того, чего в GSI нет (высокое
разрешение по тикам, позиции, история по фреймам).

## Ключевые факты про GSI (важно, не забыть)
- Предметы всех игроков приходят ТОЛЬКО в режиме спектатора/обсёрвера.
  В своей обычной игре GSI отдаёт только локального игрока (античит, by design).
- Структура в спектаторе отличается от одиночной игры.
- Раскладка: `items` -> `team2` (Radiant) / `team3` (Dire) -> `player0`..`player9`
  (player0–4 = Radiant, player5–9 = Dire). Аналогично `hero`, `player`, `abilities`.
- Названия предметов — строки в console-формате, напр. `item_black_king_bar`.
  Пустой слот = `name: "empty"`.
- Valve может молча менять ключи между патчами => код парсит защитно и дампит
  первый сырой payload в `raw_payload.json` для сверки.

## Текущее состояние кода
- `gsi_server.py` — Flask-сервер (Python). Уже работает:
  - `python gsi_server.py --make-config` — генерит .cfg для клиента Доты.
    Конфиг кладётся в `<Steam>/steamapps/common/dota 2 beta/game/dota/cfg/gamestate_integration/`.
  - `python gsi_server.py` — слушает 127.0.0.1:3000, печатает предметы по командам,
    сохраняет первый payload в `raw_payload.json`.
  - Функция `extract()` разбирает спектатор-структуру; `collect_items()` достаёт
    inventory (slot0-5), backpack (slot6-8), neutral.
- Зависимости: `flask`.
